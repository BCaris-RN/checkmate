# 007 Revisions Automation

## I. Purpose

This document governs generation of `extras/revision.txt`. The revision anchor is an operator-facing forensic summary, not a free-form prompt artifact.

## II. Generation Authority

`extras/revision.txt` is generated by:

- `scripts/build_006.py`

Policy constants used by the generator come from:

- `scripts/policy_constants.py`

## III. Dynamic Fields

The following values must be generated at runtime or sourced from current policy constants:

- Revision identifier
- Version tag
- Complexity-routing summary
- Token summary
- Human inventor name
- DraftStore compliance scan count

Hard-coded historical dates, stale version labels, and obsolete thresholds are prohibited in this document and in generated revision text.

Interpretation rule:

- Revision generation is downstream of current enforcement and current policy constants.
- `extras/revision.txt` is a generated provenance anchor, not a place to freeze example release identifiers by hand.

## IV. Required Revision Content

Every generated `extras/revision.txt` must include:

- sovereign seal header
- revision identifier
- version tag
- inventorship assertion
- routing-policy summary
- token-policy summary
- resilience and recovery summary
- DraftStore compliance summary

## V. Machine-Readable Contract Alignment

If a workflow emits a structured revision-side policy snapshot for downstream validation, it must validate against `schemas/policy_snapshot.schema.json`.

That snapshot may be projected from executable policy constants or from a validated constitutional export surface, but not from stale prose alone.

Interpretation rule:

- The schema captures shape.
- `scripts/build_006.py` and `scripts/policy_constants.py` remain the executable source of truth for generated revision content.

## VI. Operating Rule

Do not hand-author `extras/revision.txt` from an old prompt block.

Use this workflow:

1. Update policy constants intentionally when the governance model changes.
2. Run `scripts/build_006.py`.
3. Review the generated `extras/revision.txt` for release and legal correctness.
4. Preserve the file only when the build requires forensic anchoring.

## VII. Validation Rule

If the revision text conflicts with `scripts/policy_constants.py` or `scripts/build_006.py`, the executable files are the source of truth and the revision document must be regenerated.
